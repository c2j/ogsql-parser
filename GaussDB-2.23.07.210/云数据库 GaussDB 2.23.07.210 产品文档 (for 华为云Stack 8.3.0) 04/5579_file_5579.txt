动态内存使用率问题处理 动态内存使用率问题处理 故障现象 单个或者多个节点出现内存使用率超过阈值，或者内存使用率持续处于高位，或者业务执行SQL报错：ERROR:memory is temporarily unavailable。 故障影响 业务执行SQL可能报错：ERROR:memory is temporarily unavailable。 处理思路 动态内存使用率过高，一般为SQL查询数据过多或者内存参数配置不合理。 SQL查询数据过多，需要联系客户进行SQL查杀。 内存参数配置不合理，需要调整内存配置参数。 应急处理 参照 登录实例节点 ，登录出现内存问题的节点，确认当前节点占用内存高的进程。 top -M 如内存使用率高的进程为gaussdb，则继续以下步骤。 如为其他进程，如om-agent进程，则可能存在内存泄露，请联系华为技术支持。 gsql分别连接CN、主DN，查看CN、DN内存使用情况，主备版场景gsql连接主DN节点。 gsql -d postgres -p 8000 -U ${user} -W password -r select * from pg_total_memory_detail; 如上图，为当前节点CN的内存使用情况。 如上图，为当前节点DN的内存使用情况。 gsql连接DN，需要首先获取当前DN的端口号，DN的端口号可通过cm_ctl query -Cvdp命令查询获取。 基于pg_total_memory_detail视图查询结果分析。 如果max_process_memory，max_shared_memory的值设置不合理（一般一个节点上CN+DN的参数值大于总内存的5%即认为参数值设置不合理），尽快调整参数值，重设参数需要重启集群，请联系业务评估后处理。 如果dynamic_used_memory接进或者大于max_dynamic_memory时，说明当前动态内存使用较多，则执行 4 。 如果dynamic_used_memory小于max_dynamic_memory，而dynamic_peak_memory大于max_dynamic_memory表明曾经出现内存不足的情况，无需应急处理。 如果为其他情况，请联系华为技术支持。 当前动态内存使用率高处理。 动态内存使用率高，如果阻塞业务，可直接进行以下快速恢复手段 。 查询执行时间长的SQL语句。 分布式： select coorname,pid,sessionid,current_timestamp - query_start as runtime,client_addr,datname, usename, substr(query,0,100) from pgxc_stat_activity where state != 'idle' and datname!='postgres' and extract(epoch from current_timestamp-xact_start)/60 > 1 order by 1,4 desc; 主备版： select coorname,pid,sessionid,current_timestamp - query_start as runtime,client_addr,datname, usename, substr(query,0,100) from pgxc_stat_activity where state != 'idle' and datname!='$postgres' and extract(epoch from current_timestamp-xact_start)/60 > 1 order by 1,4 desc; 停止执行时间长的SQL语句会话。 分布式： execute direct on all 'select pg_terminate_session('' $pid '', '' $sessionid '');'; 主备版： select pg_terminate_session(' $pid ', ' $sessionid '); pid,sessionid由以上慢SQL查询结果获取。 pg_terminate_session 为高危操作，执行前需联系客户确认。 【快速恢复】 动态内存超过阈值，执行SQL会报错，快速恢复依赖于故障处理时间。 父主题： 资源类故障应急案例 版权所有 © 华为技术有限公司
