网络隔离，造成孤岛 网络隔离，造成孤岛 故障现象 本应急预案描述的为2AZ+仲裁AZ，4CN，4DN，4副本，9节点混合部署场景下（如图所示），AZ级网络隔离问题。 主AZ和仲裁AZ与备AZ之间网络隔离，集群被分裂成2个，其中多数派集群正常工作，查询集群状态发现集群有主，其他AZ的组件都是unknown。 业务影响 网络隔离时，强启集群可以成功，但是集群变成双主。网络恢复后，集群无主，业务无法执行，加回无法成功。 故障原因 网络隔离，造成孤岛，此时主AZ和仲裁AZ组成的小集群满足多数派，CM自动降副本后可正常工作。此时，如果忽略强启约束强制强启备AZ，则会人为造成整个集群的双主现象。当网络恢复后，加回阶段，由于主AZ降副本运行，而备AZ被强启，这时候集群便无法决策哪边的数据应该保留，集群会将两个主都降备，集群处于不可用状态，进而导致加回失败。此时需要人工干预，保留备AZ，也就是强启AZ的节点数据。 预期处理时长 强启：小于3min 加回：小于10min 日志收集要求 无。 处理思路 在AZ间网络隔离，造成网络孤岛场景，如果集群存在多数派的情况，此时多数派是可以自动的降副本+选主来提供服务的，此时非常不建议执行少数派强启。如果主AZ因为网络等问题无法提供服务，必须强启备AZ的场景，需要尽量做到以下几点来降低强启风险： 强起备AZ之前，需要停掉多数派AZ，即：停止主AZ和仲裁AZ。 尽快停掉主AZ可以降低主AZ与强启AZ同时提供服务的影响。 网络恢复后，加回之前，先启动主AZ和仲裁AZ。 注意事项 因为多数派AZ的存在，此手动流程执行结束后，因为选择强起AZ的DN为主，原多数派集群的DN会触发build操作，故而可能会出现主AZ丢失数据风险（网络孤岛，主AZ降副本之后产生的数据会丢失）。 故障定位 网络隔离后，AZ间网络不通，集群被分裂成2个，其中多数派集群正常工作，能登录后台的判断方法： 参考 登录实例节点 ，登录实例节点。 执行如下命令，查看集群状态。 cm_ctl query -Cv 发现集群有主，其他AZ的组件都是unknown。 此时认为不满足少数派强启条件。如果强启备AZ，则会手动产生双主，并且引发网络恢复后的无主问题。 网络恢复后，查看集群状态。 发现没有unknown节点(网络正常)，集群没有主机。 执行如下命令，查看ETCD的监听IP和端口。 cm_ctl view | grep etcd 执行如下命令，查看ETCD中记录的同步列表。 export ETCDCTL_API=3;
