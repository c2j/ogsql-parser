双集群switchover故障 双集群switchover故障 针对双集群下发switchover操作之后，主集群或者备集群可能返回switchover失败。本章节介绍switchover失败之后集群怎么恢复。 注意事项 switchover失败，首先尝试进行重做主从切换的任务流，如果重做无法成功，再按以下步骤确认 操作步骤 一、因为主集群（Region1）有机器异常导致的失败，此时switchover失败对主集群业务无影响。 如果需要switchover成功，则需要先修复异常的机器，再重新下发switchover。 对主备集群进行以下操作 登录core库执行以下操作，参考 10.4管控登陆core库节点指导 。 --查询对应容灾任务 SELECT * FROM CORE.DBS_DISASTER_RECOVERY WHERE ID = '9ffcab68-1b30-47dc-a7ad-d491eec22ea8'; --更新容灾任务状态 UPDATE CORE.DBS_DISASTER_RECOVERY SET UPDATED_AT= NOW (), STATUS='normal' WHERE ID = '9ffcab68-1b30-47dc-a7ad-d491eec22ea8'; 二、因为备集群（Region2）有机器异常导致的失败 可以按照如下两种方案进行恢复。 方案1：恢复到switchover之前的集群状态： 停止灾备集群 登录灾备实例节点，在灾备集群执行如下指令： cm_ctl stop 调整Dorado存储主备 在任意一端Dorado运营面，进入“数据保护”，进入“远程复制一致性组” ,按照主备实例的实例ID进行模糊搜索，找到对应实例的“远程一致性组”，如图可见当前Region2内的Dorado远程一致性组仍是“从端”。 如果Region2的Dorado远程一致性组是“主端”,则选中相应的“远程复制一致性组”，并执行“主从切换”。 如果进入的Region1的Dorado运营面，也是进行类似操作，保证Region1的Dorado远程一致性组是“主端”。 执行如下操作将原主集群重新拉起 如果旧主集群状态仍为运行中，则执行如下命令，关闭集群。 cm_ctl stop 执行如下命令，修改集群参数。 su – Ruby chroot /var/chroot source /home/Ruby/gauss_env_file gs_guc set -Z cmserver -N all -I all -c "backup_open=0" gs_guc set -Z cmagent -N all -I all -c "agent_backup_open=0" gs_guc set -Z datanode -N all -I all -c "cluster_run_mode=cluster_primary" 执行如下命令，启动集群。 cm_ctl start 4.再次执行以下命令，检查集群状态。 cm_ctl query – Cvid 如图实例状态已恢复正常。 重新拉起灾备集群 1.登录灾备实例节点。在灾备集群执行如下指令: cm_ctl start 2.再次执行以下命令，检查集群状态。 cm_ctl query – Cvid 恢复主备集群容灾任务 登录主实例和备实例所在region的Console页面。 单击左侧导航栏的“容灾管理”。 获取到对应容灾任务的id 登陆数据库core库。参考 10.4管控登陆core库节点指导 。 根据任务id对core库表进行以下操作。 a. 对备集群进行以下操作(根据备实例所在region获取到的任务id)。 --查询对应容灾任务 SELECT * FROM CORE.DBS_DISASTER_RECOVERY WHERE ID = '081499e9-1a92-4c31-a921-6b336b3724cb';
