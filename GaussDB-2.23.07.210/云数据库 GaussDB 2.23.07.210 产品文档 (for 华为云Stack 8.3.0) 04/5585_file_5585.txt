执行计划跳变应急处理 执行计划跳变应急处理 故障现象 执行计划与预期不一致，在实际执行过程中发生变化。 故障影响 执行计划跳变，可能导致SQL执行超时或者阻塞，从而影响业务阻塞。 处理思路 执行计划跳变，一般是统计信息不准确导致，现网如遇执行计划跳变，执行快速恢复步骤。 快速恢复 使用gsql连接数据库。 gsql -d postgres -p 8000 -U user -r user需替换为实际的系统管理员账号，此处需要输入密码。 分布式登录CN节点，主备版登录主DN节点。 执行以下命令查看执行计划，确认执行计划是否发生变化。 explain query ; query为目标SQL。 保存当前执行计划信息。 对目标SQL中的表执行analyze。 analyze table ; table为目标SQL中的表。 执行结束后再次查看执行计划，确认是否恢复。 explain query ; 若恢复，则快速恢复结束。 若未恢复，则继续 6 。 query为目标SQL。 开启当前会话full sql，执行当前sql语句 set track_stmt_stat_level = 'L1,L1'; select * from student where id=1; full仅开启当前会话级别，目的为方便抓取unique sql id和执行计划。其他场景不可随意开启。 获取当前sql的执行计划和unique sql id。 select unique_query_id, query, query_plan  from dbe_perf.get_global_full_sql_by_timestamp('2023-09-16 16:20:22', '2023-09-16 17:54:41') where query like '%student%'; 以上仅为示例。 对获取到的unique sql id指定Hint Patch。 select * from dbe_sql_util.create_hint_sql_patch('patch1', 186070953, 'indexscan(student)'); 以上仅为示例。 恢复确认 确认目标SQL执行是否正常。 父主题： 资源类故障应急案例 版权所有 © 华为技术有限公司
