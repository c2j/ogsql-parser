SQL限流任务详情查询、修改、删除失败 SQL限流任务详情查询、修改、删除失败 现象描述 当实例进行完备份恢复、容灾、节点替换之后，可能出现管控侧进行限流任务详情查询、修改、删除会失败。 可能原因 数据分管控层和节点内部，实例进行完备份恢复、容灾、节点替换之后，管控层原限流任务数据并没有更新绑定新的节点，导致管控层的数据和节点内部数据不一致，查询过程中查询不到数据，修改和删除也会出现失败。 定位思路 节点替换等场景出现限流任务详情查询、修改、删除失败，即可参考本章节进行处理。 处理步骤 如果是节点替换后限流任务详情查询、修改、删除失败，修复步骤如下： 在DBS运维管理平台查看节点替换任务树，在CreateNode4ReplaceContainer任务步骤里，点击操作->修改context，查看NODE_ID_LIST的属性值，记录好该节点id，为替换的node_id。 参考 登录实例节点 ，登录实例节点。 执行如下命令，连接实例。 gsql -p 8000 -d postgres -r -U root 执行如下语句，查询当前节点的限流规则。 select * from gs_workload_rule; 参考 登录core库 ，登录core。 执行如下命令，查询当前节点的限流数据。 select DBS_FLOW_LIMIT_TASK.ID, KEY_WORDS,RULE_ID from CORE.DBS_FLOW_LIMIT_TASK left join CORE.DBS_FLOW_LIMIT_TASK_RULE_INFO ON CORE.DBS_FLOW_LIMIT_TASK.ID = CORE.DBS_FLOW_LIMIT_TASK_RULE_INFO.LIMIT_TASK_ID  where CORE.DBS_FLOW_LIMIT_TASK.STATUS != 'DELETED' and CORE.DBS_FLOW_LIMIT_TASK_RULE_INFO.NODE_ID = 替换的node_id; 对比节点和core里数据，如果存在core库的key_words列值和节点option_val值（去除大括号以及转意符，冒号之后的数据）一样，且rule_id不一致，记录节点里的rule_id值和core库的ID值，在core库里执行以下语句保持一致： update  CORE.DBS_FLOW_LIMIT_TASK_RULE_INFO set RULE_ID = rule_id where ID = ID ； 举例：如1.6中查询结果有 ID 为"ae891757-c62e-48e3-bea4-33824aafe7e7"的keyword为 "asssss"，rule_id为 8，该记录在节点里的option_val值为“{asssss}”，去除括号后与keywords是一致的，但是rule_id却为7，则执行语句： update  CORE.DBS_FLOW_LIMIT_TASK_RULE_INFO set RULE_ID = '7' where ID = 'ae891757-c62e-48e3-bea4-33824aafe7e7' ； 如果是恢复到新实例、容灾倒换导致限流任务详情查询、修改、删除失败，修复步骤如下。 参考 登录core库 ，登录core。 如果是备份恢复到新实例，执行下面更新语句即可完成故障处理： update CORE.DBS_FLOW_LIMIT_TASK set INSTANCE_ID =新实例ID  where INSTANCE_ID = 原实例ID； 如果是容灾倒换，执行步骤2.3-2.6 参考 登录实例节点 ，登录主备版实例主节点，或者需要逐一登录分布式每个CN节点。 执行如下命令，连接实例。 gsql -p 8000 -d postgres -r -U root 执行如下语句，查询当前已设定的限流规则。 select * from gs_workload_rule;
