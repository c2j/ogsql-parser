pglast 失败的根因：user 和 order 是 PG 保留字

最意外的发现
MyBatis-Plus 项目最常用的两个表名 —— user 和 order —— 在 PostgreSQL 语法里是 reserved keyword，不能直接当表名用。
pglast 拒绝这些 SQL：
syntax error at or near "user", at index 12
syntax error at or near "order", at index 59
实际影响：如果你的 MyBatis-Plus 项目操作的是 PG 兼容的数据库（openGauss/GaussDB/PostgreSQL），几乎所有业务 SQL 都会卡在这俩保留字上。

pglast 失败的具体分类

失败模式 数量 示例
user 是 PG 保留字 6 INSERT INTO user (id, name) VALUES (...)
order 是 PG 保留字 3 SELECT user_id FROM order WHERE amount > 1000
LIMIT m, n MySQL 语法 1 SELECT id FROM user ORDER BY id LIMIT 20, 10
ON DUPLICATE KEY UPDATE 1 INSERT INTO user (...) ON DUPLICATE KEY UPDATE ...
? 预编译占位符 1 SELECT id FROM user WHERE deleted = 0 AND (name = ?)
其他语法差异 1 UPDATE user SET age = age + 1 ...

JSqlParser 为什么 100% 通过？

设计目标
JSqlParser 是 为 MyBatis 生态专门设计的（MyBatis-Plus 直接依赖它）。所以它对以下 MySQL-isms 都有特殊处理：
  • user / order 当作普通 identifier（不当保留字）
  • LIMIT m, n 自动识别为 offset, count
  • ON DUPLICATE KEY UPDATE 解析成 MySQL-style UPSERT
  • ? 占位符接受任意位置

实际工作流建议

基于这两个新验证，我的选择逻辑会变成
1. 静态 SQL 审计 / 数据血缘 / 性能优化：选 ogsql-parser（Rust + openGauss 原生，2.4× 速度优势）
2. MyBatis-Plus 项目做查询重写或限流：选 JSqlParser（专为 MyBatis 设计，100% 兼容）
3. PostgreSQL 数据迁移 + 业务 SQL 校验：选 sqlparser-rs（跨方言 + 快 + 不会踩 PG 保留字陷阱）
4. 跨方言 ETL 工具：选 sqlparser-rs 或 JSqlParser，避开方言专属坑
5. 别用 pglast 处理业务 SQL：除非你 100% 确定所有 SQL 都不踩 PG 保留字、只用 PG 方言
